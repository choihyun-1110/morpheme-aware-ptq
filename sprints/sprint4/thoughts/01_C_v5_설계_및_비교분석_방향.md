## 1. 논의 배경: C_v3의 한계 인식

사용자가 제기한 문제:
- C_v3 calibration set을 보니 한글 문장임에도 중국어/한자가 일부 섞여있음
- 형태소 커버리지만 보는 게 충분한가? 문장 하나하나가 "풍성"해야 calibration 효과가 클 것

핵심 통찰: **cross-sentence diversity(집합 수준)** 와 **within-sentence richness(문장 수준)** 는 다른 축이다.

```
현재 C_v3:
  128문장 집합 전체에서 Korean 형태소 커버리지 최대화
  → 집합 수준 다양성만 최적화

GPTQ가 실제로 필요한 것:
  문장 하나가 모델에 들어갔을 때 각 layer의 activation이 다양해야
  → attention Q/K/V projection의 Hessian 추정 품질
```

## 2. C_v5 알고리즘 설계

### 기존 C_v3 scoring (변경 없음)
```
score = base_diversity_score
      + α × coverage_bonus      (신규 Korean 형태소 비율)
      + β × length_bonus        (어절/토큰 길이)
      + γ × sentence_final_bonus (문장 종결형)
```

### C_v5 추가 항목: token richness bonus (δ)
```
score = base_diversity_score
      + α × coverage_bonus
      + β × length_bonus
      + γ × sentence_final_bonus
      + δ × token_richness        ← NEW
```

**token_richness = len(unique_token_ids) / len(token_ids)**

- 문장 내 unique token / 전체 token 비율
- 높을수록 반복 없이 다양한 subword로 구성된 "풍성한" 문장
- attention layer가 다양한 position/context를 처리하게 됨
- 이미 `SentenceScore.ttr`로 계산돼 있으나, diversity_score에만 반영됨
  → greedy selection scoring에 직접 추가함으로써 더 강하게 반영

### 파라미터 변경 (C_v3 → C_v5)

| 파라미터 | C_v3 | C_v5 | 이유 |
|---------|------|------|------|
| `delta` | 0.0 | **0.2** | token richness 보너스 활성화 |
| `min_ko_ratio` | 0.7 | **0.9** | 중국어/한자 bleed-through 차단 |
| `min_tokens` | 24 | **40** | 너무 짧은 문장 배제, context 보장 |

### 재현성 보장
- `delta=0.0`, `min_ko_ratio=0.7`, `min_tokens=24`로 실행하면 C_v3 동작 그대로
- C_v5는 별도 suffix로 저장됨 (`calibration_set_C_v5_*.json`)

## 3. 표준 GPTQ 대비 비교: Condition A = 표준 GPTQ

**중요한 점**: 우리의 Condition A는 GPTQ 논문의 기본 설정 그 자체.
- GPTQ 원 논문(Frantar et al., 2022): Wikitext-2 128문장을 calibration 기본값으로 사용
- 즉 A vs C_v3 비교 = **"아무 생각 없이 쓰는 표준 GPTQ" vs "우리 방법"** 비교

현재 결과 (SOLAR-10.7B KoBEST):
| 조건 | avg | boolq | copa | hellaswag | sentineg | wic |
|------|-----|-------|------|-----------|---------|-----|
| FP16 | 0.6523 | 0.8661 | 0.6400 | 0.4480 | 0.6650 | 0.5008 |
| A (표준 GPTQ) | 0.6061 | 0.7151 | 0.6160 | 0.4340 | 0.6952 | 0.4881 |
| B (랜덤 한국어) | 0.6153 | 0.7885 | 0.6300 | 0.4280 | 0.6171 | 0.4929 |
| C_v3 (우리) | 0.6250 | 0.8077 | 0.6270 | 0.4400 | 0.6322 | 0.5294 |

**C_v3이 표준 GPTQ(A) 대비 +0.0189 (avg)** → 통계 검정(S4-1)으로 유의성 확인 중.

## 4. 왜 잘 되는가: task별 분석

### boolq에서 가장 큰 개선 (+0.0926)
- boolq: 긴 한국어 지문을 읽고 yes/no 판단
- **이유**: 한국어 calibration이 긴 문장에서의 attention 패턴 학습에 유리
  - 영어(A)로 calibration하면 한국어 문장의 long-range dependency를 처리하는 Q/K matrix가 덜 최적화됨
  - 한국어는 SOV 어순 + 조사 체계 → 영어 calibration으로는 이 패턴의 Hessian을 제대로 근사 못함

### wic에서도 개선 (+0.0413)
- wic: 동일 단어가 두 문장에서 같은 의미인지 판단 (어의 중의성)
- **이유**: 한국어 다의어/동음이의어 처리에 형태소 다양성이 도움
  - C_v3는 같은 형태소의 다양한 활용형을 포함 → 의미 구분 능력 보존

### hellaswag, sentineg는 개선 미미
- hellaswag: 상식 기반 문장 완성 → 언어보다 세계 지식 의존 → calibration 언어 효과 적음
- sentineg: 감성 분석 → A가 오히려 더 좋음 (sentineg이 영어 패턴에도 robust)

### 논문에서의 해석 방향
> "형태소 다양성 기반 한국어 calibration은 한국어 reading comprehension(boolq)과 어의 판단(wic)에서 특히 유효하다. 이는 이 두 task가 언어 특화적 attention pattern에 민감하기 때문으로 해석된다."

## 5. Korean PPL (Perplexity)을 추가 지표로 활용

사용자 제안: 한국어 PPL로 언어 모델 성능을 직접 측정하자.

**의미**:
- KoBEST는 task-accuracy 기반 → 어떤 task에서 더 좋은지만 알 수 있음
- PPL은 모델의 언어 전반적 확률 분포 품질 측정
- 한국어 text에서 PPL이 낮을수록 해당 언어를 더 잘 모델링하고 있다는 의미

**일반화**: 
> "각 언어의 calibration data로 해당 언어의 PPL이 개선된다면, 그 언어에서의 LLM 능력이 실질적으로 향상된 것."

이는 논문의 contribution을 일반화하는 데 강력한 프레임:
- 한국어 데이터셋 → 한국어 PPL + KoBEST 개선
- 중국어 데이터셋 → 중국어 PPL + C-Eval 개선
- 언어 보편적 원리로 격상

### 구현 방안
```bash
# lm_eval로 PPL 측정 가능
lm_eval --model hf \
  --model_args "pretrained=...,dtype=float16" \
  --tasks ko_wiki_ppl  # 혹은 perplexity_ko
```

현재 lm-eval-harness에서 `wikitext` PPL은 기본 제공.
한국어 PPL은 KoWiki 텍스트로 직접 추가하거나, `--tasks wikitext`로 영어 PPL도 참고값으로 측정 가능.

**→ S4에 PPL 측정 task 추가 예정**

### 기대 결과
- C_v5 > C_v3: token richness + 순수 한국어 강화가 효과 있음 → within-sentence richness 가설 지지
- C_v5 ≈ C_v3: 현재 C_v3 알고리즘이 이미 충분히 좋음 → 형태소 coverage가 핵심
- C_v5 < C_v3: min_ko_ratio=0.9가 너무 엄격해서 후보 풀을 지나치게 줄임

### 논문 기여
- 두 가지 개선 방향 실증: (1) cross-sentence coverage, (2) within-sentence richness
- C_v3 대비 C_v5 ablation → 어떤 요소가 더 중요한지 분리

## 6. C_v5 결과 및 원인 분석 (2026-04-10 확인)

### C_v5 최종 결과: 0.6128 (C_v3=0.6356보다 낮음)

C_v5에서 두 변수를 동시에 바꿔서 confound 발생:
1. delta=0.2 (token richness bonus 추가)
2. min_ko_ratio=0.7→0.9 (한국어 순도 강화)
3. min_tokens=24→40 (최소 토큰 강화)

2번+3번이 unique_morphemes를 1826→1719로 줄임.
나무위키의 한자·외래어 혼합 문장들이 cross-sentence 다양성에 기여하고 있었는데, 순도 필터가 제거.
→ cross-sentence coverage 감소 → 성능 하락

## 7. C_v5_delta 실험: δ 단독 효과 분리 (2026-04-10 실행 중)

설계: C_v3 파라미터 그대로 + delta=0.2만 추가

| 실험 | delta | min_ko_ratio | min_tokens |
|------|-------|-------------|------------|
| C_v3 | 0.0 | 0.7 | 24 |
| C_v5 | 0.2 | 0.9 | 40 |
| C_v5_delta | 0.2 | 0.7 | 24 |

결과 해석:
- C_v5_delta > C_v3: delta 유효, C_v5 실패는 순도강화 탓
- C_v5_delta ~= C_v3: delta 효과 미미
- C_v5_delta < C_v3: token richness 자체가 GPTQ에 불리

스크립트: src/run_exp_c_v5_delta.sh (GPU1 실행 중)
