# 처음부터 이해하는 LLM 양자화 연구

> **대상 독자:** 양자화 논문을 읽어본 적 없는 사람  
> **목적:** 우리가 무엇을 왜 했는지, 어떤 결과가 나왔는지 처음부터 설명

---

## 1. 왜 모델을 작게 만들어야 하나

### 1.1 현재 AI 모델의 크기 문제

ChatGPT, Claude 같은 대형 언어 모델(LLM, Large Language Model)은 수십억~수천억 개의 **가중치(weight)**라는 숫자로 이루어져 있다. 가중치란 모델이 학습하면서 "이 단어 다음엔 이 단어가 잘 어울린다"는 식의 패턴을 저장한 파라미터다.

이 연구에서 주로 다룬 **SOLAR-10.7B** 모델은 가중치가 **107억 개**다.

각 가중치를 32비트 부동소수점(float32)으로 저장하면:  
```
107억 개 × 4바이트(float32) = 약 43GB
```

하지만 일반 소비자용 GPU(RTX 3090)의 VRAM은 24GB다.  
**43GB짜리 모델을 24GB GPU에 올릴 수 없다.**

### 1.2 해결책: 양자화

**양자화(Quantization)**란 각 가중치를 표현하는 데 쓰는 비트 수를 줄이는 기술이다.

```
float32 (32비트) → int8 (8비트): 메모리 1/4
float32 (32비트) → int4 (4비트): 메모리 1/8
```

SOLAR-10.7B를 4비트로 양자화하면:
```
107억 개 × 0.5바이트(4비트) = 약 5.4GB
```

24GB GPU에 올리고도 남는다.

### 1.3 문제는 정확도 손실

하지만 32비트로 저장하던 숫자를 4비트로 줄이면 **반올림 오차**가 생긴다.

```
원래 가중치: 0.31415926...
float32:     0.31415927  (거의 정확)
int4:        0.3125      (근사값)
```

이 오차가 쌓이면 모델이 틀린 답을 내거나 한국어를 이상하게 처리할 수 있다.

## 2. GPTQ: 똑똑한 4비트 양자화 방법

### 2.1 단순 반올림의 문제

가장 단순한 방법은 모든 가중치를 그냥 반올림하는 것(Round-To-Nearest, RTN)이다.  
하지만 이렇게 하면 성능이 많이 떨어진다.

왜냐하면 가중치마다 **모델 출력에 미치는 영향이 다르기** 때문이다.

어떤 가중치는 0.001 정도 오차가 생겨도 최종 출력이 거의 안 변하고,  
어떤 가중치는 0.001만 틀려도 완전히 다른 답이 나온다.

### 2.2 GPTQ의 핵심 아이디어

**GPTQ(Generative Pre-trained Transformer Quantization)**는 2022년에 나온 방법으로,  
"어떤 가중치가 얼마나 중요한지"를 **Hessian 행렬**을 이용해 계산한다.

#### Hessian이란?

Hessian은 "이 가중치를 조금 바꾸면 모델의 출력 오차가 얼마나 변하는가"를 나타내는 행렬이다.

```
Hessian의 (i,i) 대각 원소가 크다 → 가중치 i가 중요 → 조심스럽게 반올림
Hessian의 (i,i) 대각 원소가 작다 → 가중치 i가 덜 중요 → 공격적으로 반올림 가능
```

#### GPTQ 동작 과정

```
1. Calibration 데이터를 모델에 통과시킨다
2. 각 레이어의 입력값을 수집한다
3. 입력값으로 Hessian H = (1/N) Σ xᵢxᵢᵀ 를 계산한다
4. H를 보고 중요한 가중치는 살살, 덜 중요한 가중치는 과감하게 반올림한다
5. 한 가중치를 반올림한 뒤 생긴 오차를 나머지 가중치에 보상해준다
```

핵심은 3번 단계의 Hessian 계산에 쓰이는 **calibration 데이터**다.

### 2.3 Calibration 데이터의 역할

Hessian H를 계산하려면 모델에 실제 텍스트를 통과시켜야 한다.  
이때 사용하는 텍스트 샘플 128개가 바로 **calibration 데이터**다.

**비유:** 악기를 조율할 때 기준음이 필요한 것처럼,  
GPTQ는 "이 모델이 주로 어떤 입력을 처리하나"를 calibration 데이터로 파악한다.

**기존 GPTQ 논문들은 영어 텍스트(Wikitext-2, C4)를 calibration으로 사용했다.**

## 3. 우리의 연구 질문

### 3.1 발견한 문제

SOLAR-10.7B 같은 **한국어 중심 모델**을 GPTQ로 양자화할 때,  
영어 calibration 데이터를 쓰면 어떤 일이 생길까?

- 한국어 모델의 가중치는 주로 한국어 처리를 위해 조정되어 있다
- 영어 calibration을 쓰면 "한국어 처리에 중요한 가중치"를 제대로 찾지 못할 수 있다
- 그 결과, 한국어 성능이 더 많이 저하될 수 있다

### 3.2 연구 가설

> **"한국어 모델을 양자화할 때, 한국어 데이터 + 형태소 다양성이 높은 calibration을 쓰면  
> 영어 calibration보다 한국어 성능 보존이 더 잘 된다"**

---

## 4. 한국어의 특수성: 형태소

### 4.1 한국어 형태소 구조

영어는 단어 단위로 끊기지만, 한국어는 **형태소(morpheme)** 단위로 의미가 구성된다.

```
영어: "I run quickly" → 3개 단어, 의미 단위도 3개
한국어: "나는 빠르게 달린다" → 형태소로 쪼개면:
  나 + 는 (주격조사)
  빠르 + 게 (부사형 어미)  
  달리 + ㄴ + 다 (어간 + 현재시제 어미 + 종결어미)
```

AI 모델의 토크나이저(tokenizer)는 텍스트를 토큰으로 쪼개는데,  
한국어 형태소는 독립적으로 등장하거나 다른 형태소와 결합해서 수많은 조합을 만들어낸다.

### 4.2 기존 방법(C)의 문제점

초기에 단순히 "나무위키 텍스트를 랜덤으로 128개 뽑아서" calibration을 만들었더니,  
같은 단어와 형태소가 반복해서 등장하는 문제가 있었다.

예를 들어:
```
"에 대해", "을 통해", "하지만", "이라고" 
→ 이런 고빈도 조합만 계속 등장
→ 드물게 등장하는 조사, 어미, 활용형 커버 안 됨
```

**Calibration이 한국어 형태소 분포를 제대로 대표하지 못하면**,  
GPTQ가 다양한 한국어 입력에 대한 Hessian을 제대로 추정하지 못한다.

## 5. 우리의 방법: 형태소 다양성 Greedy Selection

### 5.1 알고리즘 설계 목표

128개 문장을 고를 때, **최대한 다양한 형태소 조합을 커버**하도록 선별한다.

```
목표: unique (형태소, 품사) 쌍의 수를 최대화
제약: 총 128개 문장만 사용 가능
```

### 5.2 Greedy Algorithm

```
입력: 나무위키에서 추출한 수천 개의 후보 문장
출력: 형태소 다양성이 최대인 128개 문장

알고리즘:
  1. 선택된_집합 = []
  2. 커버된_형태소 = {}
  3. 128번 반복:
     a. 아직 선택 안 된 문장 중에서
     b. "새롭게 추가되는 unique 형태소 수" 가 가장 많은 문장을 선택
     c. 선택된_집합에 추가
     d. 커버된_형태소 업데이트
  4. 선택된_집합 반환
```

### 5.3 C 버전별 개선 과정

| 버전 | 변경 사항 | 결과 |
|------|---------|------|
| C (초기) | jieba(중국어) 형태소 분석기 사용 | 한국어 분석 엉망 → 성능 하락 |
| C_v2 | KoNLPy 도입 | 개선됐지만 비한국어 subword 과다 |
| C_v3 | Mecab + 비한국어 subword 필터링 | **최고 성능** |
| C_v4 | 어절 수 추가 최적화 | C_v3보다 나쁨 (오버피팅) |

**C_v3의 핵심 변화:**  
랜덤 Korean(B)에는 비한국어 subword가 846개,  
C_v3에서는 570개로 감소 → 한국어 형태소 집중도 향상

## 6. 실험 설계

### 6.1 대상 모델

**주요 모델:** SOLAR-10.7B-Instruct-v1.0 (Upstage)
- 한국어/영어 bilingual, 사전학습 주 언어: 한국어

**검증 모델 (다모델 일반화 확인):**
- EEVE-Korean-Instruct-10.8B (yanolja): 한국어 특화
- EXAONE-3.5-7.8B (LG): 한국어+영어 혼합 사전학습
- Qwen2-7B (Alibaba): 중국어 중심 사전학습
- Llama-3-Open-Ko-8B: 영어 사전학습 + 한국어 파인튜닝

### 6.2 비교 조건

| 조건명 | 내용 | 출처 |
|--------|------|------|
| **A** | 랜덤 영어 128개 | Wikitext-2 (GPTQ 논문 기본값) |
| **B** | 랜덤 한국어 128개 | 나무위키 |
| **C_v3** | 형태소 다양성 greedy 선별 128개 | 나무위키 + Mecab |

### 6.3 양자화 설정

```
bits = 4         (4비트 양자화)
group_size = 128 (128개 가중치마다 scale 별도 계산)
desc_act = True  (중요도 순으로 처리)
알고리즘 = GPTQ
```

### 6.4 평가 지표

**KoBEST** (Korean Benchmark for Evaluation of Semantic Tasks):
- 5개 한국어 이해/추론 태스크

- 랜덤 기준선 0.5, 높을수록 좋음

**kmmlu** (Korean Massive Multitask Language Understanding):
- 57개 과목 지식 시험 (한국 수능/공무원 시험 수준)
- 4지선다, 높을수록 좋음

### 7.3 재현성 확인

C_v3 Run1=0.6354, Run2=0.6356 → **거의 동일** (표준편차 0.0001)  
같은 calibration으로 같은 모델을 두 번 양자화해도 결과가 안정적.

### 7.4 통계 검정 (5회 반복 실험)

S4-1 실험: 각 조건을 랜덤 시드를 바꾸며 5회 반복

```
C_v3 5회 결과: [0.6146, 0.6251, 0.6398, 0.6205, 0.6180]
→ 평균 0.6236,  표준편차 0.0098

B 5회 결과: [0.6309, 0.6556, 0.6676, 0.6024, 0.6402]  
→ 평균 0.6393,  표준편차 0.0250
```

**분산비 B/C_v3 = 6.46배**  
C_v3가 B보다 6배 더 안정적으로 양자화된다.

평균 차이의 t-test: p=0.1484 (5개 샘플로는 통계적 유의성 입증 어려움)  
하지만 C_v3 최솟값(0.6146) vs B 최솟값(0.6024):  
**최악의 경우에도 C_v3가 더 좋다.**

## 9. 왜 이런 결과가 나오는가: 메커니즘 분석

### 9.1 Activation 분석

GPTQ 양자화 후 각 레이어의 **활성값(activation)**을 분석했다.

활성값 = 입력 텍스트가 각 레이어를 통과할 때의 중간 출력값

```
channel_CV (채널별 변동계수) 측정:
  C_v3 calibration 으로 양자화한 모델이 가장 높음
  → 양자화 후에도 다양한 activation 패턴 유지
```

**해석:** C_v3로 학습한 Hessian이 다양한 입력 패턴을 포괄하기 때문에,  
양자화 이후에도 모델이 다양한 입력에 유연하게 반응할 수 있다.

### 9.2 Subword Coverage 분석

Calibration 데이터의 unique 토큰 커버리지 비교:

| 조건 | Unique (형태소, 품사) 쌍 | TTR (Type-Token Ratio) |
|------|----------------------|----------------------|
| A (영어) | 3,286 pairs | 0.32 |
| C_v3 (greedy 한국어) | 1,826 pairs | **0.96** |
| B (랜덤 한국어) | — | 0.87 |

C_v3의 TTR이 가장 높다 = 같은 128개 문장 안에서도 형태소 다양성이 훨씬 높다.

**A의 unique pairs가 많은 이유:** A의 문장 길이가 평균 104.9 단어로 매우 길어서  
자연스럽게 많은 단어가 포함된 것 (절대 커버리지 효과, greedy selection 효과 아님)

### 9.3 "C_en_v3 ≈ A" 실험의 통찰

영어로도 greedy selection을 적용한 C_en_v3를 만들어 영어 중심 모델(Llama3-Ko)에 실험:

- C_en_v3(0.5916) ≈ A(0.5964): 차이 미미
- **이유:** Wikitext-2(A)의 문장이 이미 평균 104.9 단어로 매우 길어서  
  greedy selection으로 선별한 짧은 문장(20 단어)보다 절대 커버리지가 더 높음

→ **Greedy selection이 효과적인 조건:**
1. 풀이 짧은 텍스트 단위로 구성되어 있을 때 (한국어 문장 단위)
2. 풀 내 품질 분산이 클 때 (나무위키: 전문/비전문 혼재)
3. 텍스트가 짧아서 한 문장에 포함되는 어휘가 제한적일 때

## 10. Task-Aware Calibration Bank (심화 연구)

### 10.1 새로운 발견

EEVE와 EXAONE35로 KoBEST(추론)와 kmmlu(지식 암기)를 함께 측정하니:

| 모델 | 태스크 | 최고 calibration |
|------|--------|----------------|
| EEVE | KoBEST (추론) | C_v3_eeve |
| EEVE | kmmlu (지식 암기) | B (랜덤 한국어) |
| EXAONE | KoBEST (추론) | B |
| EXAONE | kmmlu (지식 암기) | A (랜덤 영어) |

**한 가지 calibration으로는 두 태스크를 동시에 최적화할 수 없다!**

### 10.2 왜 이런 차이가 생기나

| Calibration | 강점 | 약점 |
|-------------|-----|-----|
| 형태소 다양성 (C_v3) | 문맥 이해·추론 패턴 커버 | 도메인 지식 커버 약함 |
| 랜덤 (B, A) | 다양한 도메인/주제 커버 | 형태소 다양성 낮음 |

### 10.3 Hybrid Calibration 실험

두 bank를 섞는 실험 진행 중:

| 조건 | C_v3 비율 | B 비율 | 방식 |
|------|---------|------|-----|
| H1 | 64개 | 64개 | 교차 배치 |
| H2 | 64개 | 64개 | C_v3 먼저 이어 붙이기 |
| H3 | 96개 | 32개 | 교차 배치 (추론 강화) |
| H4 | 32개 | 96개 | 교차 배치 (지식 강화) |

결과: balanced_retention = (KoBEST 보존율 + kmmlu 보존율) / 2 로 평가 예정

### 10.4 DBAR-v1: 더 나은 방법론 제안

Hybrid의 한계: 128개를 64+64로 나누면 각 bank의 Hessian 추정에 쓰이는 샘플이 절반

**DBAR-v1 (Dual-Bank Adaptive Rounding v1):**
```
기존 hybrid:  64개 C_v3 + 64개 B → 각 bank 샘플 절반
DBAR-v1:     128개 C_v3로 H_r 추정
             128개 B로    H_k 추정
             H_dual = λ·H_r + (1-λ)·H_k  (λ: 혼합 비율)
```

H_r과 H_k를 각각 충분한 샘플로 추정한 뒤 가중 평균으로 합침.  
λ=0.3/0.5/0.7 세 조건으로 실험 예정.

## 11. 전체 흐름 요약

```
문제 인식
  └→ 한국어 LLM을 4비트로 양자화하면 성능이 많이 떨어진다
  └→ 기존 연구는 영어 calibration만 씀

핵심 가설
  └→ 한국어 형태소 다양성이 높은 calibration이 Hessian 품질을 높여
     양자화 후 성능 보존이 더 잘 될 것이다

방법론 (C_v3)
  └→ Mecab으로 형태소 분석
  └→ Greedy selection으로 unique (형태소, 품사) 쌍 최대화
  └→ 비한국어 subword 필터링

주요 결과
  └→ SOLAR: C_v3 > B > A (KoBEST 0.6356 vs 0.5981, +3.75%p)
  └→ 5모델 중 4모델에서 "사전학습 언어 = calibration 언어" 패턴 확인
  └→ C_v3가 B보다 안정성 6.46배 높음 (분산 기준)

메커니즘
  └→ 다양한 형태소 커버 → 다양한 activation 패턴
  └→ 더 정확한 Hessian → 중요 가중치 보존
  └→ 양자화 후에도 다양한 입력에 robust

심화 연구
  └→ 태스크별 최적 calibration이 다름 발견
  └→ Hybrid/DBAR-v1: 두 bank 혼합으로 tradeoff 완화 시도
```

## 12. 한계 및 향후 연구

### 12.1 현재 한계

1. **통계적 유의성:** 5회 실험으로 p=0.1484 → n=5로는 부족. 더 많은 반복이 필요.
2. **단일 벤치마크 의존:** KoBEST는 선택형 문제. 실제 생성 품질과 다를 수 있음.
3. **모델 범위:** 4개 모델 검증. 더 다양한 아키텍처, 더 큰 모델 필요.
4. **영어 diversity의 한계:** C_en_v3 ≈ A → 텍스트 길이 효과가 diversity 효과보다 큼.

### 12.2 향후 연구 방향

1. **n 증가:** 10회 이상 반복으로 통계적 유의성 확보
2. **DBAR-v1 완성:** λ 최적값 찾기, hybrid 대비 성능 비교
3. **더 큰 모델:** 70B 이상에서도 같은 패턴이 나타나는가
4. **AWQ에도 적용:** GPTQ에만 해당되는 현상인가, 다른 양자화 방법에도 적용되는가
5. **서빙 최적화:** 양자화 후 실제 추론 속도와 메모리 사용량 프로파일링
