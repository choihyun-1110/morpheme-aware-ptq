**작성일시:** 2026-03-18
**관련 항목:** S2-2, S2-3 (베이스라인 확보)

---

## 1. 목적

기존 A/B/C/C_v2 비교는 모두 4bit GPTQ 모델 간 상대 비교였다.
FP16 원본 모델 기준 KoBEST 점수가 없으면 "어떤 조건이 얼마나 잘 보존했는가"를 절대값으로 말할 수 없다.
따라서 `upstage/SOLAR-10.7B-Instruct-v1.0` FP16 원본으로 KoBEST를 실행한다.

---

## 2. 1차 시도 및 실패

### 명령

```bash
bash src/run_eval.sh upstage/SOLAR-10.7B-Instruct-v1.0 fp16_baseline
```

### 오류

```
TypeError: LlamaForCausalLM.__init__() got an unexpected keyword argument 'dtype'
```

### 원인 분석

- `run_eval.sh`는 항상 `trust_remote_code=True`를 `model_args`에 포함
- `trust_remote_code=True`로 로드 시, SOLAR HuggingFace 허브의 커스텀 `LlamaForCausalLM`이 로드됨
- lm-evaluation-harness의 `huggingface.py:743`이 `from_pretrained(..., dtype=get_dtype(dtype), ...)`로 호출
- 이때 `dtype` kwarg가 transformers 내부에서 소비되지 않고 커스텀 `LlamaForCausalLM.__init__()`까지 전달됨
- 커스텀 `__init__`은 `dtype`을 인자로 받지 않아 TypeError 발생

### GPTQ 모델(A/B/C)이 동일 문제를 피한 이유

- GPTQ 평가 경로(`autogptq=...` 있을 때)는 `huggingface.py`의 일반 `from_pretrained` 분기가 아닌
  `AutoGPTQForCausalLM.from_quantized(...)` 경로를 탐
- 즉 `dtype=` 인자가 커스텀 LlamaForCausalLM에 닿지 않았음

---

## 3. 해결 방법

SOLAR-10.7B는 표준 Llama 아키텍처이므로 `trust_remote_code=True` 없이도 로드 가능.
`run_eval.sh` 우회 후 `lm_eval`을 직접 호출하여 `trust_remote_code` 인자를 제거.

### 실행 명령

```bash
docker exec \
    -u "$(id -u):$(id -g)" \
    -e CUDA_VISIBLE_DEVICES="0" \
    -e HOME="/home/choihyun" \
    -e HF_HOME="/home/choihyun/workspace/.cache/huggingface" \
    -e HUGGINGFACE_HUB_CACHE="/home/choihyun/workspace/.cache/huggingface/hub" \
    -e TRANSFORMERS_CACHE="/home/choihyun/workspace/.cache/huggingface/transformers" \
    llm-dev \
    bash -lc "source /opt/conda/etc/profile.d/conda.sh && conda activate llm-quant && cd /home/choihyun/workspace && lm_eval --model hf \
        --model_args 'pretrained=upstage/SOLAR-10.7B-Instruct-v1.0' \
        --tasks kobest \
        --device cuda:0 \
        --batch_size auto \
        --output_path results/eval_fp16_baseline.json \
        --log_samples"
```

출력: `results/eval_fp16_baseline.json`

---

## 4. 향후 run_eval.sh 개선 고려사항

- FP16 평가를 위해 `trust_remote_code` 없는 경로를 지원하는 옵션 플래그 추가 가능
- 또는 하네스 `huggingface.py:743`의 `dtype=` → `torch_dtype=`으로 패치 (transformers 공식 파라미터명)

---

## 4-2. 추가 트러블슈팅 - trust_remote_code 제거 후에도 동일 오류

`trust_remote_code` 없이 직접 `lm_eval` 호출해도 동일 에러 재발.

원인: 하네스의 `huggingface.py:743`이 항상 `dtype=get_dtype(dtype)` 으로 `from_pretrained`를 호출함.
transformers 4.44.2에서 이 kwarg가 모델 `__init__`까지 그대로 전달되는 것으로 확인.
올바른 파라미터명은 transformers 기준 `torch_dtype=`.

### 패치 방법

파일이 `root` 소유라 직접 수정 불가 → Docker 내부에서 root로 패치:

```bash
docker exec llm-dev bash -c "
sed -i '743s/dtype=get_dtype(dtype),/torch_dtype=get_dtype(dtype),/' \
    /home/choihyun/workspace/lm-evaluation-harness/lm_eval/models/huggingface.py
"
```

패치 후 743번 라인: `torch_dtype=get_dtype(dtype),`
