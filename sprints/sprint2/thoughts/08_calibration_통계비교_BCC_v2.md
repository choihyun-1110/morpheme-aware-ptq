## 2. 핵심 발견

### 2-1. C와 C_v2의 통계가 사실상 동일하다

모든 지표에서 C와 C_v2가 동일한 수치를 보인다.

이것이 의미하는 바:
- C_v2의 알고리즘 개정(문장형 필터 + 길이 보너스 + 문장 종결 보너스)이
  **최종 선택 문장의 통계적 분포를 실질적으로 바꾸지 못했다**
- 즉 C_v2의 성능 하락은 알고리즘 개정이 "더 나은 문장"을 선택했기 때문이 아니라,
  어떤 다른 요인(예: 후보 풀이 달라짐, 샘플링 경로 변화)에 의한 것일 가능성이 있다
- 또는 C_v2가 선택한 문장들이 C와 매우 유사한 분포를 갖고 있음에도 성능이 달랐다면,
  두 결과 차이가 단순 분산(noise)일 가능성도 있다

### 2-2. 총 고유 형태소/subword 커버리지는 C > B

| | B | C / C_v2 |
|---|---|---|
| 총 고유 형태소 | 1510 | 1934 |
| 총 고유 subword | 527 | 846 |

이전 분석 문서(`03_CondA_B_C_KoBEST_비교정리.md`)에서는
"C의 고유 형태소 커버리지(1130)가 B(1620)보다 낮다"고 기록했다.
그러나 이번 재측정에서는 C(1934)가 B(1510)보다 높게 나온다.

**측정 방법의 차이를 확인해야 한다.**
- 이전 측정: 당시 `build_calibration.py`의 메타데이터 기반 출력값이었을 가능성
- 이번 측정: Kiwi를 현재 버전으로 재실행하여 `token.form` 기준으로 집합 구성

어느 쪽이 맞든, set-level 고유 형태소 커버리지 측면에서 **C가 B보다 낮다는 기존 결론은 재검토가 필요하다**.

### 2-3. B와 C의 길이 차이는 이전 기록보다 훨씬 작다

이전 분석 문서 기록:
- B: 평균 15.49 어절, 평균 74.73 토큰
- C: 평균 5.37 어절, 평균 31.47 토큰

이번 측정:
- B: 평균 15 어절, 평균 75 토큰
- C: 평균 11 어절, 평균 70 토큰

B는 이전과 거의 동일하다. **C가 이전보다 훨씬 길게 측정된다.**

가능한 원인:
- C JSON의 `n_eojeols` 필드가 실제 text.split() 기반이 아닌 Kiwi 기반 어절 단위일 수 있음
- 이번 스크립트는 C에 저장된 `n_eojeols` 값을 그대로 사용했고,
  이전 측정은 `text.split()`으로 공백 기준 분절 수를 썼을 가능성이 있음
- 실제 텍스트를 보면 중국어/일본어 혼합 문자열이 포함된 경우
  공백 기준 어절 수와 Kiwi 기준 어절 수가 크게 달라질 수 있음

→ **어절 수 기준 통계는 측정 방법을 통일한 후 재해석해야 한다**

### 2-4. SFS 차이가 매우 크다

- B: 평균 SFS 2.30
- C: 평균 SFS 6.80 (약 3배)

SFS(Subword Fragmentation Score)는 어절당 평균 subword 토큰 수를 의미한다.
C는 어절당 약 7개의 subword 토큰으로 분절되는데, 이는 한국어 외 언어(중국어, 일본어 등)가
섞인 문장에서 tokenizer가 글자 단위로 쪼개기 때문으로 보인다.

이 점은 "C의 subword 다양성이 높다"는 해석을 지지하지만,
동시에 "C의 문장들이 자연스러운 한국어 분포에서 벗어나 있다"는 것도 시사한다.

## 3. 기존 해석에 대한 수정

### 수정 전 (03번 문서 기준)
> "C는 고유 형태소 커버리지가 B보다 낮다 → 다양성 목표가 실패했다"

### 수정 후 (이번 측정 기준)
> "C는 고유 형태소/subword 커버리지 모두 B보다 높다.
> 다만 길이(어절/토큰 수)는 B보다 짧고, SFS가 3배 높다.
> 즉 C는 '더 짧지만 더 조각난' 문장들로 구성된다."

---

## 4. 남은 해석 과제

1. **C vs B 성능 차이의 원인 재검토**
   - 길이가 원인이라면: B(평균 75토큰) > C(평균 70토큰) 차이가 실제로 크지 않음
     → 길이 단독으로는 설명이 어려울 수 있다
   - SFS가 원인이라면: 한국어 외 언어 혼합 문장이 calibration 품질을 낮출 수 있음
     → 이 가설이 더 유력해 보임

2. **C_v2 성능 하락 원인**
   - 통계가 C와 동일한데도 BoolQ가 -0.1104 하락
   - 이는 통계적으로 동일한 분포에서 뽑혔지만 **특정 문장 구성이 달랐기 때문**
   - 또는 단순 실행 분산(noise)일 가능성도 있음

3. **FP16 베이스라인과의 비교**
   - 현재 FP16 KoBEST 평가 진행 중
   - 완료 후 A/B/C의 절대적 성능 보존율 계산 예정

## 6. SFS 이상치 원인 분석: 비한국어 혼합 문장

C calibration set의 비한국어 문자(영어, 한자, 일본어, 숫자 등) 비율을 직접 측정한 결과:

| 지표 | 값 |
|------|-----|
| 평균 비한국어 비율 | 0.242 (24.2%) |
| 중앙값 비한국어 비율 | 0.241 |
| 비한국어 > 30% 문장 수 | 57 / 128 (44.5%) |

### SFS 상위 10개 문장 예시

```
SFS=14.3 | NZD, INR, EUR, JPY, PHP... 취급국가:일본,미국,...
SFS=14.0 | 던지는 포션은 '눈멀음/혼란/둔족/수면약/독약/맥주...'
SFS=12.5 | 공격력+5497/힘+87/민첩+21/의지+48/공격속도-3/크리티컬...
SFS=11.2 | 팔라완왕넓적사슴벌레(パラワンオオヒラタクワガタ) 소마의 기라파와...
SFS=10.6 | 타카쿠하테시나이요조라 ... 輝ける一番星 ... 手を伸ばして...
```

### 해석

**C의 높은 SFS는 한국어 형태소 다양성이 아니라, 비한국어 문자의 과도한 subword 분절 때문이다.**

구체적 메커니즘:
- 통화코드(NZD, JPY...), 한자(パラワン, 忘我流水道...), 일본어 가나 등은
  SOLAR tokenizer가 글자 단위 또는 UNK로 쪼개며 토큰 수가 급증
- 이 분절이 UMR과 SFS를 인위적으로 올려 다양성 점수가 높아짐
- 결과적으로 C 알고리즘이 "다양한 한국어 형태소를 가진 문장"이 아닌
  **"비한국어 문자가 많이 섞인 문장"** 을 선호하게 됨

이것이 연구 전체에서 가장 중요한 문제점이다:

> **현재 C objective의 UMR/TTR/SFS 지표가 "한국어 형태소 다양성"과 "비한국어 혼합 정도"를 구분하지 못한다.**

이 문제는 BoolQ 하락을 설명하는 핵심 가설로도 연결된다:
- 비한국어 혼합 문장으로 calibration하면 GPTQ가 한국어 자연 분포보다
  "글자 단위 token embedding"에 치우친 Hessian 추정값을 갖게 될 수 있음
- 이 경우 BoolQ처럼 자연 한국어 문맥 이해가 필요한 태스크에서 성능이 떨어질 가능성이 있음
