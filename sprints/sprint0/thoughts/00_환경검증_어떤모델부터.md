## 3. 내 생각

### ⚠️ 메모리 추정치에 대한 주의

아래 "FP16 크기"는 **모델 가중치(weight)만의 크기**이다:
- 계산: 파라미터 수 × 2 bytes (FP16) = 가중치 크기
- **실제 양자화 과정에서는 이것보다 더 필요함:**
  - Calibration 시 activation 텐서 저장
  - GPTQ: Hessian 역행렬 근사 계산 → 레이어당 추가 메모리
  - AWQ: scale 탐색 과정 → 추가 메모리
  - 임시 버퍼, optimizer state 등
- **이 오버헤드가 정확히 얼마인지는 돌려봐야만 안다**
- → 그래서 파일럿 실행 시 `nvidia-smi`로 peak memory를 반드시 측정해야 함

### TITAN RTX 24GB 단일 GPU에서 가능한가?

- **EXAONE-3.5-7.8B:** 가중치 ~15.6GB + 오버헤드 → 24GB 내 **여유 ~8GB**, 안정적
- **Llama-3.1-8B:** 가중치 ~16GB + 오버헤드 → 비슷한 수준, 가능
- **SOLAR-10.7B:** 가중치 ~21.4GB + 오버헤드 → 24GB **초과 가능성 높음** ⚠️

### GPU 2장 합쳐서 48GB처럼 쓸 수 있는가?

**결론: 모델 로드는 가능, 양자화 calibration은 제한적**

| 방법 | 설명 | 양자화에서? |
|------|------|-------------|
| `device_map="auto"` | HuggingFace가 레이어를 GPU 0/1에 자동 분배 | ✅ 로드 가능, calibration은 라이브러리 의존 |
| Tensor Parallelism | 하나의 레이어를 2장에 쪼개서 병렬 연산 | ❌ GPTQ/AWQ 미지원 |
| NVLink 메모리 합산 | GPU 간 메모리를 물리적으로 합산 | ❌ TITAN RTX는 NVLink 미지원 |

**현실적 시나리오:**
- `device_map="auto"`로 SOLAR-10.7B를 2장에 나눠 올리는 건 **가능**
- 하지만 `auto-gptq`나 `autoawq`가 **multi-GPU calibration을 제대로 지원하는지는 확인 필요**
  - 지원하면 → SOLAR도 양자화 가능 (48GB 활용)
  - 미지원이면 → 단일 GPU 24GB 제한, SOLAR는 OOM 위험
- **→ 파일럿(S0-6)에서 확인할 항목에 추가**

### 어떤 모델부터?

**EXAONE-3.5-7.8B를 첫 파일럿으로 추천하는 이유:**
1. 한국어 학습 비중이 가장 높음 → 우리 가설(형태소 다양성 효과)이 가장 잘 드러날 모델
2. 단일 GPU에서도 메모리 여유 → 실험 실패 위험 낮음
3. 양자화 성공 시 → 바로 Llama-3.1과 비교 가능 (같은 7~8B급)
4. 이 파일럿에서 peak memory를 측정하면, SOLAR 가능 여부도 역산 가능

### SOLAR-10.7B는 어떻게 할까?

**선택지:**
- **(A)** 일단 제외, EXAONE + Llama 2개 모델로 진행 → 시간 절약, 비교도 충분
- **(B)** 파일럿 후 multi-GPU 양자화 지원 여부 확인 → 되면 포함, 안 되면 제외
- **(C)** SOLAR 대신 더 작은 한국어 모델 대체 (예: polyglot-ko-5.8B 등)
