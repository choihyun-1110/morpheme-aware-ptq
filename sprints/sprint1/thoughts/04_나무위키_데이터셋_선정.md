## 2. 선정: `heegyu/namuwiki-extracted` ✅

### 선정 이유

1. **전처리 완료 상태**
   - 위키 마크업(헤더, 테이블, 특수 태그)이 이미 제거되어 있어, 우리가 추가로 해야 할 전처리 작업량이 대폭 감소.
   - raw dump를 쓰면 `== 개요 ==` 같은 마크업이 형태소 분석기에 노이즈로 작용할 수 있음.

2. **검증된 데이터**
   - HuggingFace에서 8개 한국어 LLM(kogpt-j 등)의 사전학습에 실제로 사용된 이력.
   - 데이터 품질이 실전 검증됨.

3. **적절한 규모**
   - 565K 문서에서 문장 분리 시 **수백만 문장** 확보 가능.
   - 10만 문장 후보 풀 구성에 충분히 넉넉함.

4. **재현성**
   - HuggingFace에 영구 호스팅되어 있으므로, 논문에서 `load_dataset("heegyu/namuwiki-extracted")`만 명시하면 누구나 동일 데이터로 재현 가능.
   - 직접 크롤링 시 시점별 데이터 차이로 재현 불가능해질 위험.

5. **사용 편의성**
   - `pip install datasets` + 한 줄 코드로 즉시 로드.
   - 별도 크롤러/파서 개발 불필요.

### 주의사항
- 덤프 시점: **2022년 3월** → 최신 문서는 미포함. 단, 우리 연구는 "형태소 다양성"에 집중하므로 시점은 큰 문제 아님.
- `[math(...)]` 태그가 일부 미처리된 경우 있음 → 전처리 시 정규식으로 추가 제거 필요.
- 나무위키 라이선스(CC BY-NC-SA 2.0) 확인 필요 — 학술 연구 목적이므로 문제 없을 것으로 예상.

## 3. 데이터 전처리 파이프라인 (우리가 추가로 해야 할 것)

```
heegyu/namuwiki-extracted (565K 문서)
  → 문장 분리 (kss 또는 정규식 기반)
  → 필터링:
      - 20자 미만 / 500자 초과 문장 제거
      - 한국어 비율 50% 미만 문장 제거
      - [math(...)] 등 잔여 마크업 제거
  → 랜덤 샘플링으로 10만 문장 후보 풀 구성
  → Kiwi 형태소 분석 + Tokenizer 적용
  → 다양성 점수 산출 → Greedy Selection
  → 128문장 calibration set 출력
```
