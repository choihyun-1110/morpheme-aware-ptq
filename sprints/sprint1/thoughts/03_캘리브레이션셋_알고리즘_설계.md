## 1. 목적 (Problem Statement)

PTQ(Post-Training Quantization)에서 calibration 데이터를 구성할 때, 기존 방식은 **영어 Wikitext-2에서 128문장을 랜덤 샘플링**하는 것이 표준이다.

본 연구는 한국어(교착어)의 형태소 다양성을 최대화하는 문장을 **전략적으로 선별**하여 calibration set를 구성하고, 이것이 양자화 품질에 어떤 영향을 미치는지 검증한다.

**이 문서에서 설계할 것:**
- 나무위키 코퍼스에서 "형태소 다양성이 최대인 128문장"을 추출하는 **구체적 알고리즘**
- 형태소(Kiwi) ↔ LLM subword tokenizer 간의 **정렬(alignment) 전략**
- 4가지 실험 조건(A/B/C/D)의 데이터 구축 파이프라인

  - 순수 한국어 비율이 낮은 문장 (영어/숫자 비율 > 50%) 필터링

## 3. 핵심 알고리즘: 다양성 점수(Diversity Score) 산출

### 3-1. 세 가지 지표

각 문장 $s$ 에 대해 다음 세 지표를 산출:

#### ① Unique Morpheme Ratio (UMR)
```
UMR(s) = |{고유 (형태소, 품사) 쌍}| / |전체 형태소 수|
```
- **측정 도구:** Kiwi 형태소 분석기
- **의미:** 같은 형태소가 반복되지 않고 다양한 형태소가 등장하는 정도
- **예시:**
  - "나는 나를 나에게 보냈다" → UMR 낮음 (나 반복)
  - "철수가 학교에서 과학을 공부했다" → UMR 높음 (각기 다른 형태소)

#### ② Type-Token Ratio (TTR) — LLM Tokenizer 기준
```
TTR(s) = |{고유 subword 토큰}| / |전체 subword 토큰 수|
```
- **측정 도구:** 대상 모델의 tokenizer (`AutoTokenizer`)
- **의미:** LLM이 실제로 바라보는 토큰 단위에서의 다양성
- **핵심:** Solar(vocab 32K)와 Llama 3(vocab 128K)에서 **서로 다른 TTR 값**이 나올 수 있음 → 모델별로 다양성 점수가 달라짐

#### ③ Subword Fragmentation Score (SFS)
```
SFS(s) = (전체 subword 토큰 수) / (공백 기준 어절 수)
```
- **의미:** 한 어절이 평균 몇 개의 subword로 분절되는지. 높을수록 tokenizer가 해당 문장을 세밀하게 쪼개야 하므로, activation 분포가 다양해질 가능성.
- **예상:** 한국어 교착어 특성상 조사/어미 결합이 복잡한 문장에서 SFS가 높아짐.

### 3-2. 최종 다양성 점수 (Composite Diversity Score)

```
D(s) = w₁ × norm(UMR) + w₂ × norm(TTR) + w₃ × norm(SFS)
```

- `norm(·)`: min-max normalization (전체 코퍼스 기준 0~1 스케일링)
- 초기 가중치: **w₁ = w₂ = w₃ = 1/3** (균등 가중)

> ⚠️ **가중치 결정 전략:**
> 1단계: 균등 가중(1:1:1)으로 우선 실험 → 결과 확인
> 이후 ablation study로 가중치 변화에 따른 성능 차이 분석 가능 (Sprint 3 이후)

### 3-3. 정규화 vs 비정규화 설계 결정

**min-max 정규화를 적용하는 이유:**
- UMR은 0~1 범위지만, SFS는 1~10+ 범위가 될 수 있음
- 스케일이 다른 지표를 동등하게 합산하려면 정규화 필수
- 전체 후보 코퍼스(수만 문장)에 대해 한 번 계산 후 정규화 수행

## 4. 문장 선택 알고리즘: Greedy Diversity Selection

### 4-1. 왜 단순 Top-K가 아닌 Greedy인가?

단순히 D(s) 상위 128문장을 뽑으면, 특정 도메인/어휘에 편중될 수 있다.
예: 법률 텍스트만 다양성이 높아서 상위 128개가 전부 법률 관련이 될 위험.

**Greedy selection은 이미 선택된 문장과의 중복을 페널티로 주어 커버리지를 극대화한다.**

### 4-2. 알고리즘 의사코드

```python
def greedy_diversity_select(candidates, n=128):
    """
    candidates: List[Dict] - 각 원소에 text, morphemes, tokens, scores 포함
    n: 선택할 문장 수
    """
    selected = []
    covered_morphemes = set()  # 이미 커버된 (형태소, 품사) 쌍 집합
    
    for _ in range(n):
        best_score = -inf
        best_idx = -1
        
        for i, cand in enumerate(candidates):
            if cand in selected:
                continue
            
            # 기본 다양성 점수
            base_score = cand['diversity_score']
            
            # 신규 형태소 커버리지 보너스
            new_morphemes = cand['morpheme_set'] - covered_morphemes
            coverage_bonus = len(new_morphemes) / (len(cand['morpheme_set']) + 1e-8)
            
            # 최종 선택 점수 = 다양성 + 커버리지 보너스
            final_score = base_score + α * coverage_bonus
            
            if final_score > best_score:
                best_score = final_score
                best_idx = i
        
        selected.append(candidates[best_idx])
        covered_morphemes |= candidates[best_idx]['morpheme_set']
    
    return selected
```

- **α (커버리지 보너스 계수):** 초기값 `0.3` 설정.
  - α가 높으면 → 새로운 형태소를 많이 담은 문장 우선 (커버리지 중시)
  - α가 낮으면 → 개별 문장의 다양성 점수를 더 중시
- **시간 복잡도:** O(n × |candidates|) — candidates가 수만 개여도 128번 루프라 수 초 이내 완료 예상.

### 4-3. 대안: MMR (Maximal Marginal Relevance) 스타일

추후 coverage_bonus 대신 문장 간 cosine similarity를 사용한 MMR 방식도 고려 가능.
단, 현재 단계에서는 형태소 집합 기반의 set coverage가 더 직관적이고 해석 가능하므로 Greedy 방식 우선 채택.

## 5. 형태소-토큰 정렬(Alignment) 전략

### 5-1. 문제 정의

형태소 분석기(Kiwi)와 LLM tokenizer(BPE/SentencePiece)는 **서로 다른 기준**으로 텍스트를 분절한다.

```
원문:        "양자화라이브러리를사용해봅시다"
Kiwi 형태소:  양자화 / 라이브러리 / 를 / 사용 / 하 / 어 / 보 / ㅂ시다
Solar 토큰:   ▁양 / 자 / 화 / 라이 / 브러 / 리를 / 사용 / 해 / 봅 / 시다
```

→ 두 분절 경계가 일치하지 않으므로, "이 토큰이 어떤 형태소에 속하는가?"를 판단하는 정렬이 필요.

### 5-2. 정렬 방법: Character-level Span Mapping

```python
def align_morphemes_to_tokens(text, kiwi_result, tokenizer):
    """
    1. Kiwi 결과에서 각 형태소의 문자 단위 시작/끝 위치(char span)를 추출
    2. Tokenizer 결과에서 각 토큰의 문자 단위 시작/끝 위치를 추출
    3. 두 span 집합의 overlap을 계산하여 mapping 생성
    """
    # Step 1: 형태소 → char span
    morph_spans = []
    for token in kiwi_result:
        morph_spans.append({
            'form': token.form,
            'tag': token.tag,
            'start': token.start,
            'end': token.start + token.len
        })
    
    # Step 2: subword 토큰 → char span
    encoding = tokenizer(text, return_offsets_mapping=True)
    token_spans = encoding['offset_mapping']  # [(start, end), ...]
    
    # Step 3: 각 subword 토큰이 어떤 형태소(들)에 걸치는지 매핑
    alignment = []
    for tok_idx, (tok_start, tok_end) in enumerate(token_spans):
        overlapping_morphs = []
        for morph in morph_spans:
            if tok_start < morph['end'] and tok_end > morph['start']:
                overlapping_morphs.append(morph['tag'])
        alignment.append({
            'token_id': encoding['input_ids'][tok_idx],
            'token_text': tokenizer.decode([encoding['input_ids'][tok_idx]]),
            'morpheme_tags': overlapping_morphs
        })
    
    return alignment
```

### 5-3. 정렬 결과의 활용 (분석 단계)

이 alignment 정보는 **S1-2 알고리즘 자체에서는 직접 사용하지 않지만**, Sprint 2~3의 activation 분석 시 핵심적으로 사용됨:
- "특정 품사(예: 조사 `-를`, `-에서`)가 activation outlier를 유발하는가?"
- "형태소 경계와 subword 경계가 불일치하는 토큰 위치에서 양자화 에러가 집중되는가?"

→ **현 단계에서는 alignment 함수를 구현해두되, 실험은 Sprint 2에서 수행.**

## 6. 4가지 실험 조건 데이터 구축 파이프라인

### 전체 파이프라인 개요

```
[나무위키 원본]
    → 전처리 (HTML 제거, 길이 필터링)
    → Kiwi 형태소 분석
    → LLM Tokenizer 적용
    → 다양성 점수 산출
    → Greedy Selection (조건 C)
    → 128문장 calibration set 출력
```

### 조건별 데이터 구축

| 조건 | 소스 | 선택 방법 | 문장 수 |
|------|------|-----------|---------|
| **A. 영어 기준** | Wikitext-2 | 랜덤 샘플링 | 128 |
| **B. 한국어 랜덤** | 나무위키 | 랜덤 샘플링 | 128 |
| **C. 한국어 형태소 다양성 Max** | 나무위키 | **Greedy Diversity Selection** | 128 |
| **D. 혼합** | Wikitext-2 + 나무위키 | 영어 랜덤 64 + 한국어 다양성 64 | 128 |

### 조건 C의 세부 단계

1. 나무위키 전처리 → 후보 문장 풀 구성 (예: 10만 문장)
2. 각 문장에 Kiwi 형태소 분석 실행 → UMR 산출
3. 각 문장에 대상 모델 tokenizer 적용 → TTR, SFS 산출
4. min-max 정규화 후 D(s) 계산
5. Greedy Diversity Selection으로 128문장 추출
6. 추출된 문장을 tokenizer로 인코딩하여 `List[Dict]` 형태로 저장

> 💡 **모델별 분기 포인트:** TTR과 SFS는 모델 tokenizer에 의존하므로, Solar용 조건 C와 Llama 3 Ko용 조건 C는 **다른 128문장**이 선택될 수 있음. 이 차이 자체도 분석 대상이 됨.

## 8. 미결 사항 & 다음 단계

### 확정 사항 ✅
- 형태소 분석기: **Kiwi (kiwipiepy)**
- 알고리즘: **Greedy Diversity Selection** with 3-indicator composite score
- 초기 가중치: **균등 (1:1:1)**
- 커버리지 보너스 α: **0.3**
- 문장 수: **128문장** (기존 표준과 동일, 공정 비교)
- 한국어 데이터 소스: **`heegyu/namuwiki-extracted`** (HuggingFace 공개 덤프, 565K 문서, 전처리 완료)
  - 선정 상세: [04_나무위키_데이터셋_선정.md](04_나무위키_데이터셋_선정.md)
- 후보 풀 크기: **10만 문장** (565K 문서에서 문장 분리 후 랜덤 샘플링)
- 모델별 조건 C 전략: **2단계(Phase) 접근**
  - Phase 1: **공통 128문장** (UMR 기반, tokenizer 무관)으로 Solar/Llama 3 공정 비교
  - Phase 2 (여력 시): **모델별 최적화 128문장** (TTR/SFS 포함)으로 추가 실험 → tokenizer-aware selection의 효과 검증
